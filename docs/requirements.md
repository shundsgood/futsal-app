# フットサルチーム管理Webアプリ 要件定義書

## 1. プロジェクト概要

フットサルチームの運営に必要な以下の情報を一元管理するWebアプリを開発する。

* 日程調整
* イベント・スケジュール管理
* 出欠管理
* メンバー管理
* 試合結果管理
* 得点・アシスト管理
* 個人成績・チーム成績の可視化
* ユニフォーム情報管理

本アプリは、LINEのスケジュール機能やスプレッドシートなどに分散している情報を、一つのWebアプリ上で管理できることを目的とする。

---

# 2. アプリの基本方針

## 2.1 操作負担を減らす

以下を基本方針とする。

* 同じ情報を何度も入力させない
* 登録済みメンバーを各機能で再利用する
* 日程調整から正式イベントをシームレスに作成する
* 出欠情報を別画面で再入力させない
* 得点者やアシスト者を手入力させない
* イベント参加者を試合記録に再利用する
* 同一イベント内の試合結果を連続登録できるようにする

## 2.2 チーム単位で管理する

予定、メンバー、試合結果、成績などはすべてチーム単位で管理する。

1ユーザーが複数チームに所属できる構成とする。

---

# 3. 利用者区分

## 3.1 オーナー

チームを作成したユーザー。

権限は以下とする。

* チーム情報の編集
* チーム削除
* 管理者の設定
* メンバーの追加・編集
* 日程調整の作成・編集・削除
* イベントの作成・編集・削除
* 試合結果の登録・編集・削除
* 個人成績の修正

## 3.2 管理者

チーム運営を行うユーザー。

権限は以下とする。

* メンバー管理
* 日程調整管理
* イベント管理
* 出欠確認
* 試合結果登録
* 得点・アシスト登録
* 成績確認

## 3.3 一般メンバー

チームに所属するメンバー。

権限は以下とする。

* 日程調整への回答
* イベントへの出欠回答
* スケジュール確認
* 試合結果確認
* 個人成績・ランキング確認
* 自分のプロフィール情報編集

# 5. チーム管理機能

## 5.1 チーム作成

ユーザーは新しいチームを作成できる。

必須項目：

* チーム名

任意項目：

* チームロゴ
* 活動エリア
* チーム説明

チーム作成時、作成者をオーナーとして登録する。

## 5.2 チーム招待

管理者は招待URLを発行できる。

メンバーは招待URLからチームに参加できる。

## 5.3 チーム切り替え

複数チームに所属するユーザーは、現在表示するチームを切り替えられる。

# 6. メンバー管理機能

## 6.1 メンバー情報

メンバーごとに以下を管理する。

必須項目：

* 表示名
* 在籍ステータス

任意項目：

* 背番号
* ポジション
* ユニフォームサイズ
* 入団日
* プロフィール画像
* 備考

在籍ステータス：

* 在籍中
* 休止中
* 退団済み

## 6.2 削除方針

過去の試合記録や個人成績を維持するため、メンバーは原則として物理削除しない。

退団時は「退団済み」に変更する。

退団済みメンバーは、新規の得点者・アシスト者選択候補には原則表示しない。

過去の試合記録には引き続き表示する。

## 6.3 メンバー情報の共通利用

登録済みメンバー情報は以下の機能で共通利用する。

* 日程調整回答
* 出欠回答
* 出場メンバー登録
* 得点者選択
* アシスト者選択
* 得点ランキング
* アシストランキング
* 個人プロフィール
* ユニフォーム情報

名前や背番号を各機能で再入力させない。

# 7. 日程調整機能

## 7.1 日程調整作成

管理者は、複数の候補日時を設定して日程調整を作成できる。

登録項目：

* タイトル
* 説明・備考
* 回答期限
* 候補日時
* 候補ごとの開始時刻
* 候補ごとの終了時刻
* 候補ごとの会場
* 候補ごとの補足
* イベント種別

イベント種別：

* 練習
* 練習試合
* 大会
* リーグ戦
* その他

候補日時は複数登録できる。

## 7.2 日程調整回答

メンバーは候補日時ごとに以下を回答できる。

* ○：参加可能
* △：未定・条件付き
* ×：参加不可

任意でコメントを入力できる。

回答者名はログイン中ユーザーに紐づく登録済みメンバーから自動判定する。

回答のたびに名前を入力させない。

## 7.3 回答一覧

管理者およびメンバーは、候補日時ごとの回答状況を確認できる。

表示項目：

* ○の人数
* △の人数
* ×の人数
* 未回答者数
* 回答者一覧
* 回答コメント

表示形式は、候補日別の集計表示およびメンバー別マトリクス表示に対応する。

## 7.4 候補日の並び替え

候補日は以下の条件で並び替え可能とする。

* 日時順
* ○の人数が多い順
* ○＋△の人数が多い順

---

# 8. 日程調整からイベントへのシームレスな変換

## 8.1 基本仕様

管理者は、日程調整の候補日から1つを選択し、そのまま正式イベントとして確定できる。

管理者が「この日程で確定」を実行した場合、別画面で同じ内容を再入力させない。

以下の情報を日程調整から正式イベントへ自動で引き継ぐ。

* タイトル
* イベント種別
* 説明・備考
* 確定した日時
* 開始時刻
* 終了時刻
* 会場
* 候補日の補足
* 日程調整の回答情報

## 8.2 確定操作フロー

日程調整詳細画面に、候補日ごとに以下のボタンを表示する。

```text
この日程で確定
```

押下後、確認画面または確認モーダルを表示する。

確認項目：

* イベント名
* 日時
* 会場
* イベント種別
* 回答引き継ぎ内容

管理者は必要な場合のみ情報を修正できる。

確定後、正式イベントを自動作成する。

## 8.3 出欠情報の自動変換

日程調整の回答は、イベントの出欠情報に以下のルールで変換する。

* ○ → 参加
* △ → 未定
* × → 不参加
* 未回答 → 未回答

イベント作成後、メンバーは改めて出欠を入力し直す必要がない。

イベント確定後は、必要に応じて各メンバーが出欠情報を変更できる。

## 8.4 確定後の日程調整状態

イベント作成後、元の日程調整は「確定済み」状態に変更する。

ステータス：

* 回答受付中
* 回答締切
* 確定済み
* 中止

確定済みの日程調整画面には、作成された正式イベントへのリンクを表示する。

```text
この日程調整からイベントが作成されました
［イベント詳細を見る］
```

## 8.5 二重作成防止

同じ日程調整から複数の正式イベントが誤って作成されないようにする。

原則として、1つの日程調整から作成できる正式イベントは1件とする。

すでにイベントが作成済みの場合は、再度「確定」ボタンを表示しない。

## 8.6 確定後の変更

正式イベント作成後に日時や会場を変更する場合は、正式イベント側で編集する。

元の日程調整の候補日時を変更しても、作成済みイベントには自動反映しない。

ただし、日程調整画面から作成済みイベントに遷移できるようにする。

## 8.7 キャンセル時

確定済みイベントを削除または中止しても、元の日程調整の回答データは保持する。

管理者は必要に応じて以下を選択できる。

* イベントのみ中止する
* 日程調整を再度回答受付中に戻す
* 新しい日程調整を複製して作成する

登録項目：

* イベント種別

* 開始時刻
* 終了時刻
* 集合時刻
* 会場

## 9.2 イベント種別

* 練習
* 練習試合
* 大会

## 9.3 出欠回答

# 10. 試合結果管理機能

## 10.1 イベントとの紐付け

試合結果は、練習試合、大会、リーグ戦などのイベントに紐付けて登録する。

1つのイベントに複数試合を登録できる。

例：

```text
6月20日 フットサル大会
├─ 第1試合
├─ 第2試合
└─ 第3試合
```

## 10.2 試合結果登録項目

* 対戦相手名
* 自チーム得点
* 相手チーム得点
* 試合順
* 試合開始時刻
* 試合時間
* メモ
* 出場メンバー

勝敗はスコアから自動判定する。

* 自チーム得点 > 相手得点：勝利
* 自チーム得点 = 相手得点：引き分け
* 自チーム得点 < 相手得点：敗戦

## 10.3 複数試合の連続登録

大会や練習試合で複数試合を登録しやすいようにする。

以下の操作を用意する。

* 登録して次の試合を入力
* 前の試合の出場メンバーをコピー
* 対戦相手名だけ変更して続けて登録
* イベント情報は自動保持

---

# 11. 出場メンバー管理

## 11.1 初期設定

イベントの参加回答が「参加」のメンバーを、試合の出場メンバー候補として自動表示する。

試合作成時は、参加者を初期状態で選択済みにする。

管理者は出場していないメンバーだけチェックを外せる。

## 11.2 同一イベント内のコピー

同一イベント内に複数試合がある場合、前の試合の出場メンバーをコピーできる。

```text
［前の試合の出場メンバーをコピー］
```

## 11.3 出場試合数

出場メンバーとして登録された試合を、そのメンバーの出場試合数として集計する。

# 12. 得点・アシスト登録機能

## 12.1 基本仕様

得点者およびアシスト者は、チームに登録済みのメンバーから選択する。

名前の自由入力は原則不可とする。

選択候補には以下を表示する。

* 背番号
* 表示名

## 12.2 得点登録項目

* 得点者
* アシスト者
* 得点種別
* 備考

得点種別：

* 通常得点
* オウンゴール
* 得点者不明

アシスト者は以下も選択可能とする。

* アシストなし
* 不明

## 12.3 選択候補

基本的には、その試合の出場メンバーを優先的に表示する。

出場メンバーに登録されていない選手も、必要に応じてチームメンバー一覧から選択できる。

## 12.4 入力負担軽減

複数得点を連続で登録できるようにする。

ボタン例：

```text
［登録して次の得点を入力］
［登録して終了］
```

登録後も同じ試合画面を維持する。

最近選択したメンバーを候補上部に表示する。

名前検索および背番号検索に対応する。

## 12.5 スコアとの整合性

登録した得点記録数と自チーム得点数が一致しない場合、警告を表示する。

例：

```text
試合スコアは3得点ですが、得点記録は2件です。
```

ただし、得点者不明やオウンゴールを登録できるため、保存自体は禁止しない。

# 13. 成績・ランキング機能

## 13.1 個人成績

メンバーごとに以下を集計する。

* 出場試合数
* 得点数
* アシスト数
* ゴール関与数
* 1試合平均得点
* 1試合平均アシスト
* 1試合平均ゴール関与
* イベント参加回数

ゴール関与数は以下で計算する。

```text
ゴール関与数 = 得点数 + アシスト数
```

## 13.2 ランキング

以下のランキングを表示する。

* 得点ランキング
* アシストランキング
* ゴール関与ランキング
* 出場試合数ランキング

表示項目：

* 順位
* メンバー名
* 背番号
* 出場試合数
* 得点
* アシスト
* ゴール関与数

## 13.3 チーム成績

以下を集計する。

* 試合数
* 勝利数
* 引き分け数
* 敗戦数
* 勝率
* 総得点
* 総失点
* 得失点差
* 1試合平均得点
* 1試合平均失点

勝率は以下で計算する。

```text
勝率 = 勝利数 ÷ 試合数
```

## 13.4 絞り込み

以下の条件で成績を絞り込める。

* 年
* 期間
* イベント種別
* 大会・イベント
* 通算

# 15. ユニフォーム情報

# 20. 主要ユースケース

## 日程調整からイベント作成

1. 管理者が日程調整を作成する
2. 候補日時を複数登録する
3. メンバーが○△×で回答する
4. 管理者が回答状況を確認する
5. 候補日を1つ選択する
6. 「この日程で確定」を押す
7. 日程調整の内容を引き継いだイベントが自動作成される
8. ○△×が参加・未定・不参加に変換される
9. イベント詳細画面に自動遷移する
10. メンバーは必要な場合のみ出欠を修正する

## 試合結果から個人成績反映

1. 管理者がイベント詳細を開く
2. 試合結果を追加する
3. イベント参加者が出場メンバーとして初期選択される
4. 必要に応じて出場者を修正する
5. 得点者を登録済みメンバーから選択する
6. アシスト者を登録済みメンバーから選択する
7. 複数得点を連続登録する
8. 試合結果を保存する
9. チーム成績に自動反映する
10. 個人成績・ランキングに自動反映する

# 21. UI・UX要件

* スマートフォンでの利用を前提とする
* LINEから共有URLを開いて操作しやすくする
* 主要操作は3タップ程度で完了できることを目標とする
* 日程調整の回答は1画面で完結させる
* 得点登録はモーダルまたはインライン入力とする
* メンバー選択は検索可能なプルダウンとする
* 背番号と名前を併記する
* 日程調整確定後はイベント詳細へ自動遷移する
* 同じ情報の再入力をできる限り避ける
* 削除操作には確認ダイアログを表示する
* 保存完了後は明確なフィードバックを表示する

# 22. 非機能要件

* レスポンシブ対応
* スマートフォン、タブレット、PCで利用可能
* 通信はHTTPSを使用
* チームごとにデータアクセスを分離する
* 他チームの情報は閲覧できない
* 管理者権限をサーバー側でも検証する
* 入力値バリデーションを行う
* 試合結果および成績データの整合性を維持する
* メンバー退団後も過去記録を保持する
* 日程調整からのイベント二重作成を防止する
* 日程確定処理はトランザクションで実行する

---

# 23. 実装上の重要事項

日程調整からイベントを確定する処理では、以下を1つのトランザクションとして実行する。

1. 対象の日程調整が未確定であることを確認
2. 選択された候補日時を取得
3. 正式イベントを作成
4. 日程調整回答をイベント出欠へ変換
5. 元の日程調整を確定済みに更新
6. confirmed_option_idを保存
7. created_event_idを保存

途中でエラーが発生した場合は、すべてロールバックする。

同じ日程調整からイベントが複数作成されないよう、以下の制約を設ける。

* schedule_polls.created_event_idに一意制約を設定
* 確定処理時にステータスを確認
* サーバー側で二重送信を防止
* 確定ボタン押下後はボタンを無効化
